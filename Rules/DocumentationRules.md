# Правила Документування (Documentation Rules)

[← На Головну](Main.md)

Цей документ визначає стандарти написання технічної документації для проекту Merge2.

## 1. Загальні Принципи
- **Мова**: Українська.
- **Стиль**: Лаконічний, технічний, структурований.
- **Формат**: Markdown.

## 2. Структура Документів
Кожен документ повинен мати чітку структуру:
- **Навігація**: Посилання [← На Головну](Main.md) або `../Main.md` на самому початку.
- **Заголовок (# Title)**: Назва компонента або системи.
- **Опис**: Коротке пояснення, що це і для чого потрібно.
- **Основні Властивості**: Список ключових параметрів або характеристик.
- **Взаємодія**: Опис того, як цей компонент взаємодіє з іншими частинами системи.
- **Візуальні Ефекти (Optional)**: Опис візуального супроводу.

## 3. Специфічні Правила

### 3.1. Документація Чіпів (Chips)
Для кожного типу чіпа (наприклад, `ChipGenerator`, `ChipContainer`) має бути окремий файл у папці `Docs/Chips/`.
- **Назва Файлу**: Має відповідати назві класу (наприклад, `ChipGenerator.md`).
- **Унікальні Ефекти**: Якщо чіп має унікальні візуальні ефекти, вони описуються в цьому ж документі.
    > **Важливо**: В загальному документі `Docs/Effects.md` має бути посилання на цей документ, а не дублювання опису.

### 3.2. Посилання
- Використовуйте відносні посилання на інші документи (наприклад, `[Tech Design](../Technical/TechnicalDesign.md)`).
- Посилайтеся на код, де це можливо.

### 3.3. Структура Компонентів
Описуйте компоненти через їхні класи та конкретне призначення, уникаючи абстрактних архітектурних термінів (View, Controller, Model).
- **Архітектура та Відповідальність**:
  - **`ClassName.cs`**: Короткий опис, що робить цей клас.
  - **`DataClassName.cs`**: Опис налаштувань.
  - **`LogicClassName.cs`**: Опис винесеної логіки.
- **Процес (Flow)**: Покроковий опис технічного флоу (Input -> Method -> Event).

### 3.4. Технічний Дизайн (Technical Design)
Документ `Docs/Technical/TechnicalDesign.md` є центральним місцем опису архітектури.
- **Вимога**: Цей документ повинен описувати всі інтерфейси з папки `Modules/Merge2/Scripts/VContainer/Interfaces`.
- **Вміст**: Для кожного інтерфейсу необхідно вказати його призначення та конкретну реалізацію (клас).

### 3.5. Документація Тестів (Testing)
Документ `Docs/Technical/Testing.md` описує підхід до тестування та тестову інфраструктуру.
- **Вимога**: Документ повинен містити опис **кожного тесту** з тестових класів.
- **Структура опису тесту**:
  - Назва тестового методу.
  - Що саме перевіряє тест (мета тесту).
  - Які умови/сценарії покриває.
- **Оновлення**: При додаванні нового тестового методу обов'язково додається його опис у `Testing.md`.

## 4. Підтримка та Актуалізація (Maintenance)
Документація повинна бути "живою" і відповідати поточному стану коду.

### Коли оновлювати
- **Logic Change**: При зміні логіки роботи компонента (наприклад, алгоритм спавну).
- **Data Extention**: При додаванні нових полів у дані (наприклад, нові параметри в `ChipData`).
- **Refactoring**: При зміні архітектурних підходів або інтерфейсів.

### Що саме оновлювати
1. **Локальні файли** (`Docs/Chips/`, `Docs/Features/`):
   - Оновлюються завжди при змінах у відповідних класах.
   - Якщо додається новий функціонал — додається новий розділ.

2. **Technical Design** (`Docs/Technical/TechnicalDesign.md`):
   - Обов'язкове оновлення при додаванні/зміні **інтерфейсів** у `VContainer/Interfaces`.
   - При зміні графу залежностей (LifetimeScope).

3. **Main.md**:
   - Оновлюється **тільки** при додаванні нових `.md` файлів, щоб додати посилання в зміст.
